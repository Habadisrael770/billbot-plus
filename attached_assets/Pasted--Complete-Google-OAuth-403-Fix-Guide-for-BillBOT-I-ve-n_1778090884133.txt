## **STEP 2 — Medium Term (Avoid Gmail Verification)**

Here are your **4 realistic options** to avoid needing Google's security review for `gmail.readonly`:

### **Option A: Service Account + Domain-Wide Delegation ❌ (Not ideal for SaaS)**
- **How it works**: Your backend uses a service account to access Gmail on behalf of users
- **Pros**: No Google verification needed, no restricted scopes
- **Cons**: 
  - Requires organization domain setup (doesn't work for individual Gmail accounts)
  - Users need to configure domain-wide delegation in Google Admin
  - Too complicated for a SaaS targeting regular Gmail users
- **Verdict**: Skip this — it's enterprise-only

### **Option B: Use IMAP/SMTP instead of Gmail API ✅ (RECOMMENDED SHORT-TERM)**
- **How it works**: Use IMAP to scan emails instead of Gmail API
- **Pros**: 
  - No OAuth scopes verification needed
  - Works with any email provider (Gmail, Outlook, etc.)
  - Simple for users: they generate an "App Password" in Gmail settings
  - No API keys or OAuth nightmare
- **Cons**: 
  - Slightly slower than Gmail API
  - IMAP protocol is older but very stable
- **Verdict**: **USE THIS as your medium-term approach**
- **User flow**: 
  - User clicks "Connect Email"
  - You guide them to generate Gmail App Password (2-factor required)
  - They paste the App Password into your app
  - You connect via IMAP — instant, no OAuth
- **Setup**: Use Node.js `imap` or `mail-parser` libraries

### **Option C: Use a Different Gmail Scope (Non-Restricted) ❌ (Limited value)**
- Scopes like `gmail.modify`, `gmail.send`, etc. are also restricted
- The problem is **restricted scope = Google verification required**
- Only `email` and `profile` scopes don't need verification, but they don't give you inbox access
- **Verdict**: Not practical for invoice scanning

### **Option D: Get Google Verification (Long-Term) ✅ (Do this later)**
- Eventually you'll want official OAuth for better UX

```javascript

```
- ✅ Using `'*'` is fine for same-origin (popup → parent iframe)

- For testing: `'*'` is okay, but fix before production
